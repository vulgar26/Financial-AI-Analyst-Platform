# VISION — 项目定位

> 本文是定位基准。其它文档与本文冲突时，以本文为准。
> 最后重定：2026-06-13（经一整场需求重谈，推翻了旧的「价值投资研究工作台/只读不给建议」定位）。

## 一句话定位

一个**我自己用、也用来练技术的金融学习与分析系统**。它的第一价值是工程：用业界成熟框架做出一个架构清晰、有真实数据接入、有 AI 能力、能跑给人看的完整系统；金融是让我有动力、并顺带接触真实数据与指标的题材。

## 双目标与优先级（这是判断一切的尺子）

1. **第一目标：技术成长 / 简历价值。** 很多技术决策优先考虑工程价值，而不是「能不能直接帮我赚钱」。我是学计算机的工程师，这个项目要能写进简历、面试讲得透。
2. **第二目标：金融认知成长。** 我确实想学投资和金融分析，希望这个项目成为我**接触真实金融数据、理解财务指标、建立分析框架**的载体。但我清楚真正学投资不靠一个项目——那走项目之外的路（读书、问通用 AI、以后小额实盘）。

两个目标都在，只是优先级不同。**金融不是随便选的题材，但它在本项目里是「题材和副产品」，不是「目的」。**

## 评判功能/技术决策的判据

> 看「**技术含量 / 简历价值 / 工程是否加分**」，不看「对我投资有没有用」。
> 更一般地：判断要不要引入某技术/框架，不看「用不用得上」，看「用了之后**净效果加分还是减分**」——不碍事就可以用。

由此推出的具体倾向：

- **成熟框架放手用、放手换。** 用 Spring AI 模块化 RAG 替换自研检索、用业界标准重构自研实现——这正是简历核心卖点（「用业界标准重构了自研 X」是面试硬货）。门槛低到「乐意 + 不碍事」即可。
- **真实数据 + 工具治理是宝。** 接第三方数据源、做熔断/降级/限流，是简历上拿得出手的东西，做厚。
- **纯噱头、技术上没含量的功能不做。** 例如「买卖点提示」——既帮不了我（我判断不了对错），技术上也没含量，砍。
- **自研复杂机制门槛高。** 只在「产品/学习真需要，且没有现成替代」时才自己造。

## 被这次重定推翻的旧设定（避免重蹈）

旧文档（已删的 PRODUCT_VISION.md、finance-observer-design.md 等）共有一种「保守人格」：
**只读 / 不选股 / 不给买卖建议 / 帮你想清楚不替你决策 / 砍合规风险。**

这套设定是为「做产品给别人用」设计的（担责、合规、黑箱信任）。但本项目确认是**自用 + 简历项目**，那三大顾虑不成立。所以这套保守线**不再是定位基准**。这不代表要乱来——合规免责声明作为工程上的良好实践仍保留（见 ARCHITECTURE 安全章节），但它不再是「砍功能」的理由。

> 历史教训仍然有效的那一条：**别先搭漂亮框架再填内容（空地基陷阱）。** 想做高级的东西，就用成熟框架最小落地、真跑起来，而不是再写一版设计文档。

## 现有工程基础如何对口

过去的工程积累不是白做，它们是这个系统的真实零件（详见 ARCHITECTURE）：

| 已有能力 | 角色 |
| --- | --- |
| `PLAN → RETRIEVE → TOOL → GUARD → WRITE` 固定链路 + workflow runtime | 可解释的决策流水线（简历核心） |
| 真实基本面数据接入（FMP）+ 工具治理 | 真实数据 + 熔断/降级/限流（简历硬货） |
| eval 两层评测体系 | 可评测、可回归（简历亮点） |
| RAG（Spring AI 官方 PgVectorStore + 自研编排） | 检索增强；编排部分是后续换框架的候选 |
| 异步任务框架骨架 | 未来主动巡检/报告的基础（远期） |
